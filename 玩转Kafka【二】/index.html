玩转Kafka【二】

发表于2022-09-14|更新于2022-09-14|Kafka
|字数总计:2.8k|阅读时长:11分钟|阅读量:

上节内容,我们已经使用 Docker 将 Zookeeper 集群与 Kafka 集群搭建好,这节内容我们就来演示一下如何使用 Kafka,以及Kafka的一些源码分析。

Kafka Controller

Controller是 Apache Kafka 的核心组件。它的主要作用是在 Apache ZooKeeper 的帮助下管理和协调整个 Kafka 集群。集群中任意一台 Broker 都能充当控制器的角色，但是，在运行过程中，只能有一个 Broker 成为控制器，行使其管理和协调的职责。换句话说，每个正常运转的 Kafka 集群，在任意时刻都有且只有一个控制器。官网上有个名为 activeController 的 JMX 指标，可以帮助我们实时监控控制器的存活状态。这个 JMX 指标非常关键，在实际运维操作过程中，一定要实时查看这个指标的值。

Controller 是如何被选出来的

实际上,集群中的每台Broker启动时,都会向Zookeeper上创建一个/controller的节点,而第一个创建成功的Broker则为Leader。当作为Leader的Broker发生故障后,其余的Follower会再次竞争去创建/controller的节点,从而选出新的Leader。但在实际应用中,很可能由于网络波动问题,让Kafka误以为Leader挂掉了,导致产生新一轮选举,此时会出现同时存在两个Leader,剩下的Follower就不知道该听谁的了。这就是一个典型的脑裂问题。

Kafka如何解决脑裂问题

Kafka在每个Leader产生的时候,会在Zookeeper中生成一个更大的/controller_epoch,并同步给所有的Follower。这样,原epoch较小的Leader发送的命令会被其余Follower忽略掉。

Kafka脚本及源码分析

  • 进入Kafka的容器内部

    docker exec -it kafka1 /bin/bash

  • 跳转Kafka的bin目录,cd /opt/bitnami/kafka/bin可以看到很多shell脚本

  • 这些shell脚本就是封装了一系列用于操作kafka的命令,接下来我们一起演示一下kafka常用的一些操作,并跟着源码一起分析具体流程。

创建Topic

我们创建一个Topic,副本为3,分区数为5,名称为FirstTopic

这时就有个问题了,我们集群有3台Broker,5个Partition和三个副本,那么Kafka是如何管理这些Partition和副本的呢？每个分区的Leader是怎么选出来的呢？我们一起来看一下源码,打开kafka-topics.sh文件,内容很简单,只有一条命令:exec $(dirname $0)/kafka-run-class.sh kafka.admin.TopicCommand "$@",可以看到是调用的TopicCommand类的main方法,我们跟踪代码可找到assignReplicasToBrokersRackUnaware方法。

当分区和副本确定后,Kafka会默认设置每个分区第一个副本为Leader。我们可以使用describe命令查看一下是否与我们分析的一致。

查看Topic描述信息

可以看出,与我们分析代码完全一致,每个分区的副本都被均匀散布在3台Broker上。

删除Topic

删除命令执行完,会发现控制台打印了两句话:

提示我们Topic已经被标记删除了,但是并没有立即删除,并且需要开启delete.topic.enable=true的配置,才可以删除Topic。当然这个delete.topic.enable=true是默认开启的,这个我们在kafka启动的时候可以看到。

那么这个Topic是怎么从标记删除到真正被Kafka删除的呢？同样一起来看一下源码,进入TopicCommand类的main方法,这次跟踪的是 topicService.deleteTopic(opts)

可以看出Kafka是在ZK的/admin/delete_topics 目录下创建了一个topicName的节点,也就是所谓的标记删除。那么真正删除的操作在哪呢？从Kafka在ZK中创建了一个节点,很容易猜到是使用了ZK的监听器,监听admin/delete-topics下是否存在数据, 具体操作我们可以在源码中找到TopicDeletionManager类,从类名就能看出来这是一个管理删除Topic的类。
